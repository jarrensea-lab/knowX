## 迭代记录格式

每条记录包含:
- **版本**: 对应 CHANGELOG 版本号
- **优化目标**: 这次迭代要解决什么问题
- **方案选择**: 为什么选择这个方案 (而不是其他方案)
- **实际效果**: 优化后的量化效果
- **副作用**: 引入的新问题
- **下一步**: 后续优化方向

## 迭代记录

### v2.3.0 — 多模型分工

**版本**: 2.3.0
**日期**: 2026-05-26
**优化目标**: 单模型 35B 处理所有角色导致速度慢 (~6min/次分析)
**方案选择**: 
  - 方案 A: 统一用 9B 模型 (快但质量下降)
  - 方案 B: 按角色复杂度分配不同模型 (选定方案)
  - 方案 C: 云 API (成本高，数据需上传)
**决策依据**: 利用 Ollama 2-model 内存限制，将简单任务(风控/估值)交给 9B，复杂任务(短线分析)保留 35B
**实际效果**: 盘前 ~6min → ~3min (50%提升)，盘中 ~3min → ~1.5min
**副作用**: 35B 和 R1 不能同时驻留，裁判需切换模型 (~30s)
**下一步**: 评估是否引入更小的模型 (1.5B) 做风控，进一步释放大模型

### v2.2.0 — 异步生成

**版本**: 2.2.0
**日期**: 2026-05-26
**优化目标**: AI 分析 2-3 分钟，前端 30s 超时，导致功能完全不可用
**方案选择**: FastAPI BackgroundTasks + 前端轮询
**实际效果**: 解决了超时问题，用户可正常使用 AI 分析
**副作用**: 用户需等待 2-3 分钟，体验仍不够好
**下一步**: 考虑 WebSocket 推送完成通知，替代轮询

---

### v2.0.0 — 多源容错

**版本**: 2.0.0
**日期**: 2026-05-26
**优化目标**: 单一数据源故障导致整个系统不可用
**方案选择**: 3 数据源 (腾讯+新浪+东财) + 断路器熔断 + 分层缓存
**实际效果**: 任意 2 个数据源故障系统仍可工作
**副作用**: 代码复杂度增加 (router + cache + 3 个 client)
**下一步**: mootdx 兼容 Python 3.14 后接入作为首选源
